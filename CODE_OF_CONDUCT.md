# Code of Conduct

This project is for nonprofits. That shapes the behaviour expected here.

## What we expect

- **Treat nonprofit staff asking questions as the primary audience.** They are not a nuisance or a distraction — they are the reason this repo exists. Answer plainly. No condescension.
- **Assume good faith.** Most people opening issues are tired, under-resourced, and short on time. Give them a useful reply before asking clarifying questions.
- **Arabic, Urdu, Tagalog, Malayalam, Bengali, Hindi, Amharic, and English** are all fine for issues and comments. If you reply, reply in a language the original poster can read. Machine translation is better than silence.
- **Disagree in public, decide in the open.** If two contributors disagree on an approach, argue it in the issue. Document the outcome in TECH_MEMORY.md or DEFENSE_BRIEF.md so someone revisiting the decision later can understand why it landed where it did.
- **Credit contributors.** When a skill is merged, add the contributor's GitHub handle to the skill's frontmatter under `metadata.author`. This is not optional.

## What we will not tolerate

- Discrimination, harassment, or personal attacks of any kind.
- Promoting paid services, consulting offers, or closed platforms in issues or PRs. If you have a commercial offering that might help, link it in a separate comment — don't frame the issue itself as marketing.
- Sharing real beneficiary data, donor PII, or NGO internal information in examples or test fixtures. Ever. Even if it looks harmless, don't.
- Claiming credit for work you did not do.

## Reporting

If someone crosses a line, open an issue tagged `conduct` (if public is appropriate) or email the maintainer directly via the contact link on the maintainer's GitHub profile.

## Enforcement

The maintainer will:

1. Acknowledge the report within seven days.
2. Investigate what was said, in what context, and whether it was a first or repeat pattern.
3. Take one of: no action with reasoning given, private note to the person, public warning, temporary ban, permanent ban.
4. Record the outcome in a private maintainer log — not in the public repo — unless the resolution itself is educational and worth publishing (e.g., a clarification of what is and isn't in scope).

This document is deliberately short. If it becomes clear it needs to grow, it will.
